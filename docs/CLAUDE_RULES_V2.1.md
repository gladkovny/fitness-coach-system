## 📐 ОГРАНИЧЕНИЕ СЛОЖНОСТИ ЗАДАЧ

**Сложные задачи делить на маленькие этапы и делать изменения постепенно.**

- Не брать в одну сессию крупный рефакторинг или много файлов сразу — перегружает среду (процессор/Cursor), выше риск ошибок.
- Сложную задачу разбить на шаги: сначала план этапов, затем по одному этапу за раз, после каждого — проверка (и при необходимости коммит).
- Один этап = одна логическая порция (один файл или одна функция/фича), чтобы можно было проверить и при необходимости откатить.
- Если задача большая — предложить разбиение на подзадачи и спросить, с какого этапа начать.

### [4] AUTHENTICATION — Вход и доступ

**Что это:** Как пользователи входят в систему.

**Текущий статус: ✅ НАСТРОЕНО (Supabase Auth)**

```
Настроено:
├── Supabase Auth (email + пароль)
├── Тренер привязан к auth.users через auth_id
├── RLS политики по auth.uid()
└── Страница входа: deploy/master/login.html

Роли:
├── Trainer (тренер)     — видит своих клиентов
└── Client (клиент)      — будущее (пока нет входа для клиентов)
```

**Как войти:**
1. Открыть `http://localhost:3000/login.html`
2. Email: тот что в Supabase Auth
3. Пароль: установленный при создании пользователя

**Риски при изменениях:** 🔴 Высокий
**⚠️ КРИТИЧНО:** Не менять RLS политики без понимания последствий

**Что это:** Защита данных пользователей.

**Текущий статус: ⚠️ БАЗОВЫЙ УРОВЕНЬ**

```
Реализовано:
├── Supabase Auth (шифрование паролей)
```

- НИКОГДА не хранить пароли в коде
- НИКОГДА не коммитить .env файлы
- НИКОГДА не логировать токены
