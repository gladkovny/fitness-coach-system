# Роадмап проекта: Reference Architecture и приоритет «единая система для моих клиентов»

> **Приоритет:** построить одну систему для текущих клиентов (Ярослав, Кирилл, Марк, Алёна). Масштабирование (другие тренеры, биллинг, мультитенантность) — только после того, как система полностью внедрена в твою работу.
>
> **Визуализация архитектуры:** [docs/excalidraw-saas-reference-architecture.excalidraw.json](excalidraw-saas-reference-architecture.excalidraw.json) — открыть в [excalidraw.com](https://excalidraw.com) (Open).

---

## 1. Reference Architecture — сопоставление с текущим состоянием

Архитектура из 6 слоёв (см. диаграмму «SaaS Reference Architecture — Full Stack») сопоставлена с тем, что уже есть в проекте.

| Слой | В Reference Architecture | Сейчас в проекте | Статус |
|------|--------------------------|-------------------|--------|
| **1. Frontend** | Trainer Web, Trainer Mobile (PWA), Client Web/Mobile, Telegram Bot, Landing | Trainer: login, cabinet, dashboard, tracker (Vanilla JS). Client: дашборд Марка (GAS), программа Марка (GAS). Telegram/Landing — нет. | ✅ Тренерский контур на Supabase. ⏳ Марк — отдельный контур (GAS). |
| **2. API + Auth** | API Gateway, Auth (роли), Webhooks, File upload | Supabase Auth (email/password), запросы с фронта напрямую к Supabase (RLS). Отдельного gateway нет. | ✅ Auth и RLS работают. Gateway/ webhooks — не нужны для текущих клиентов. |
| **3. Business Logic** | Training, Nutrition, Progress, Communication, Billing, Onboarding, Gamification | Training (программы, блоки, сессии, подходы) — в Supabase. Progress — дашборд. Onboarding — форма → Assessment (в работе). Nutrition — daily_logs. Billing, Communication, Gamification — нет. | ✅ Ядро тренировок и прогресса. ⏳ Онбординг допиливаем. Остальное — не в приоритете. |
| **4. Integration Hub** | Wearables, Telegram, Stripe, AI, OCR, аналитика | Edge Function parse-workout (AI распознавание упражнений). Остальные интеграции — нет. | ✅ AI для тренировок. Whoop/Telegram/Stripe — отложены. |
| **5. Data Layer** | PostgreSQL, Redis, Queue, Storage, Search | PostgreSQL (Supabase). Storage — по мере надобности. Redis/Queue/Search — нет. | ✅ БД достаточно для одного тренера и 4 клиентов. |
| **6. Infrastructure** | CI/CD, мониторинг, логи, бэкапы, CDN | Netlify (деплой), Supabase (хостинг БД и Auth), GitHub. Pre-commit (lint, тесты). Бэкапы — ручные/по необходимости. | ✅ Минимум для стабильной работы. |

**Вывод:** для цели «единая система для текущих клиентов» не хватает в первую очередь **объединения контуров** (Марк перевести на Supabase) и **стабилизации** (баги, онбординг Алёны). Остальные слои Reference Architecture можно наращивать позже или не трогать до этапа масштабирования.

## 2. Цель этапа: единая система для текущих клиентов

**Смысл:** один тренер (ты), один бэкенд (Supabase), все четыре клиента обслуживаются из одной системы. Нет двух контуров (Master vs Mark) — один контур данных и один набор страниц для тебя и для клиентов.

**Текущие клиенты:**

| Клиент | Тип | Сейчас | Целевое состояние |
|--------|-----|--------|-------------------|
| Ярослав | offline, сплит | Supabase (дашборд, трекер) | Оставить как есть |
| Кирилл | offline, фулбоди | Supabase (дашборд, трекер) | Оставить как есть, фикс багов по фидбеку |
| Марк | online, 90 дней | Дашборд и программа на GAS | Дашборд и программа на Supabase (один контур) |
| Алёна | hybrid | Онбординг (форма → Assessment) | После онбординга — в той же системе, что и остальные |

**Критерий готовности этапа:** ты ведёшь всех четырёх клиентов через один вход (login) и один бэкенд (Supabase); дашборд и программа Марка не зависят от GAS; онбординг Алёны закрыт и воспроизводим.

## 3. План по этапам (до масштабирования)

### Этап A: Стабилизация (сейчас)

**Цель:** текущие клиенты работают без сбоев, онбординг Алёны доведён до конца.

| # | Задача | Приоритет | Результат |
|---|--------|-----------|-----------|
| A1 | Фикс багов по фидбеку Ярослава и Кирилла | 🔴 | Трекер и дашборд предсказуемы в использовании |
| A2 | Тестирование Unified Tracker v4.4 (2 недели) | 🔴 | Уверенность в трекере перед объединением контуров |
| A3 | Онбординг Алёны: Form → Assessment, автозаполнение Goals из формы | 🔴 | Алёна в системе, цикл онбординга проверен |
| A4 | Бэкап данных (Supabase + Sheets при необходимости) | 🔴 | Безопасность перед следующими шагами |
| A5 | Документация текущего состояния (при необходимости) | 🟡 | SYNC_STATUS и роадмап актуальны |

**Задачи из «на будущее» (по мере времени):** интенсивность при bodyweight (BUG_BODYWEIGHT_INTENSITY), обязательные задачи для Кирилла (mandatory_tasks), страница персональных данных клиента.

---

### Этап B: Единый контур — Марк на Supabase

**Цель:** дашборд и программа Марка работают на данных из Supabase. Контур Mark (GAS) не используется для Марка.

| # | Задача | Приоритет | Результат |
|---|--------|-----------|-----------|
| B1 | Дашборд Марка на Supabase: данные (блок, недели, прогресс) из Supabase вместо GAS | 🔴 | deploy/mark/dashboard переключён на Supabase или заменён страницей в deploy/master |
| B2 | Программа/трекер Марка (день X/90): план на день и логирование из Supabase | 🔴 | Один трекер/программа для онлайн-клиента в той же БД, что и офлайн |
| B3 | Отключение/архивация GAS для Марка после проверки | 🟡 | Нет зависимости от двух бэкендов |

**Важно:** не обязательно переписывать всё на React сразу — достаточно, чтобы Марк обслуживался из Supabase (текущий стек Vanilla + Supabase допустим для этапа «единая система для своих»).

---

### Этап C: Полировка под свою работу

**Цель:** система удобна лично тебе в ежедневной работе; мелкие улучшения, а не новые большие фичи.

| # | Задача | Приоритет | Результат |
|---|--------|-----------|-----------|
| C1 | Улучшения трекера и дашборда по твоему опыту использования | 🟡 | Меньше трений в работе |
| C2 | Опционально: офлайн/ local-first в трекере (очередь подходов при отсутствии сети) | 🟢 | Удобство в зале без стабильного интернета |
| C3 | Единая точка входа: кабинет как старт (все клиенты, быстрые ссылки на дашборд/трекер) | 🟡 | Один экран «мои клиенты» и действия |

Переход на React PWA и дизайн-систему Olympus (из ARCHITECTURE_V2) можно делать в этом этапе или отложить — в зависимости от того, насколько текущий Vanilla-интерфейс тебя устраивает после стабилизации и переноса Марка.

## 4. Отложено: масштабирование (после полного внедрения у себя)

**Старт этого блока только после того, как ты полностью внедришь систему в свою работу** (все 4 клиента стабильно в одной системе, ты ею пользуешься без обходных путей).

| Направление | Что входит | Когда рассматривать |
|-------------|------------|----------------------|
| Другие тренеры | Мультитенантность, отдельные аккаунты тренеров, лимиты по тарифам | После 2–4 недель стабильной работы «только для своих» |
| Биллинг | Stripe Connect, подписки, тарифы | Вместе с выходом на других тренеров |
| Партнёрская программа | Список партнёров, оффер бета-тренерам, презентация | После принятия решения о масштабировании |
| Интеграции для маркетинга | Telegram-бот, Whoop, Apple Health, InBody OCR | По потребности клиентов или при масштабировании |
| Геймификация, AI-питание | Достижения, серии, разбор еды по фото | После стабильного ядра и по желанию |
